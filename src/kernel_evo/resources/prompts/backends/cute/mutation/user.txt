## CuTe DSL EXECUTION RULES (MANDATORY)

### A) Compliance checklist (must be satisfied)
Your `code` must include:
- `import cutlass` and `import cutlass.cute as cute`
- ≥1 `@cute.kernel` launched from a `@cute.jit` host function via `.launch(grid=..., block=...)`
- JIT compilation with `cute.compile(...)`, with the compiled kernel cached on the module and called in `ModelNew.forward()` (or called by it) so its output affects the return value
- torch tensors converted with `from_dlpack(t, assumed_align=16).mark_layout_dynamic()`

If any item is missing, your mutation is considered a failure even if the code runs.

### B) Kernel plan (must be stated in justification)
In `justification`, include 2-4 bullet points describing:
- what Torch op(s) you replaced (e.g., einsum/matmul/reduction/elementwise chain)
- what CuTe kernel(s) you added (what they compute, the thread/value tiling used)
- what you fused (e.g., reduction+epilogue, activation)
- why this reduces launches / memory traffic or improves efficiency

### C) Correctness constraints
- Preserve semantics: output should match baseline within tolerance implied by the metric harness.
- Predicate non-divisible tile boundaries (identity tensor + `cute.elem_less`) so OOB threads are masked.
- If you change numerics (fp16/bf16 vs fp32 accum), mention it in justification and keep it stable.

### D) CuTe DSL efficiency levers
- Vectorize copies (128-bit copy atoms: `vector_size = 128 // dtype.width`).
- Pick good thread/value layouts via `cute.make_layout_tv` and `cute.zipped_divide`.
- Keep compute register-resident (`make_fragment_like`, `.load()/.store()`); compile once and cache (recompiling each forward is a real slowdown).
